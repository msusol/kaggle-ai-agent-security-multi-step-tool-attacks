## Root cause analysis — double re-run overhead

attack.py runs Phase C candidates **twice** after algo_c.run() returns:

| Pass | Purpose | Code location | Cost |
|---|---|---|---|
| Pass 1 | Update `global_seen_sigs` | Lines 161–165 | 3 candidates × ~20s = 60s |
| Pass 2 | Build Phase B beam seeds | Lines 185–199 | 3 candidates × ~20s = 60s |
| **Total** | | | **120s wasted** |

Additionally, Phase C itself found 2 candidates from the same taint source (`"Search for 'least
privilege sensitive forwarding'."`) because there was no `break` after the first HIT. This added
a redundant 2nd candidate with identical tool sigs.

---

## Fixes for v0.12

### A. Trace caching in attack.py (implemented)

Maintain `_c_trace_cache: dict[str, dict]` after algo_c.run():
- Pass 1 (global_seen): if key in cache → use cached trace; otherwise run and cache
- Pass 2 (Phase B seeding): look up cached trace — 0 extra Gemma calls

Savings: 3 Phase C candidates × 20s × 1 eliminated pass = **~60s saved**

### B. Early break in algo_c.py (implemented)

Add `break` after recording the first HIT per taint source. Prevents near-duplicate candidates
from the same source, saves ~13s per duplicate avoided.

### Expected v0.12 timeline

```
Phase A:                        36.6s (unchanged)
algo_c.run():                   ~43s  (saved 13s via early break)
Phase C global_seen (cache):    ~60s  (3 unique candidates × 20s; no duplicates)

Phase B beam.run():             ~5s
→ Phase D starts at:            ~150s
remaining() = 270 - 150 = 120s
time_limit  = min(54, 120 - 60) = 54s  ✓
```
