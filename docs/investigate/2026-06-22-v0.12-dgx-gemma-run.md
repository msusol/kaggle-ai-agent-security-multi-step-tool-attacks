## Root cause analysis — Phase C re-runs still happening

v0.12 had two fixes applied vs v0.11:

| Fix | Status | Effect |
|---|---|---|
| `break` after first HIT per taint source (algo_c.py) | ✓ Applied | Eliminated duplicate candidate from same source |
| Cache Phase B seeding (attack.py, `_c_trace_cache`) | ✓ Applied | Phase B seeding uses cached traces → 0 re-runs for seeding |
| trace_store to eliminate global_seen re-runs | ✗ Not yet | global_seen update still re-runs Phase C candidates |

The global_seen update loop (attack.py lines 163-174) checked `_c_trace_cache` for existing
entries, but in v0.12 the cache was only populated AFTER running algo_c (by the Phase B seeding
code), not during. So the first pass (global_seen update) always missed the cache and re-ran.

Only pass 2 (Phase B seeding) benefited from the cache — saving ~40s vs v0.11. But pass 1
still cost ~40s, leaving Phase D with only 1.8s.

---

## Fixes for v0.13

### trace_store parameter (implemented)

Pass `trace_store: dict | None = None` to `algo_c.run()`. algo_c populates it at each HIT time:
```python
if trace_store is not None:
    trace_store[str(steps)] = trace
```

attack.py passes `_c_trace_cache` as `trace_store`. Since algo_c writes the trace before
returning each HIT, the global_seen update loop finds entries in cache on first access.
**Effect: global_seen update costs 0 extra Gemma calls.**

### Phase B budget reservation (implemented)

Phase B time_limit now subtracts `budget_for("D")` explicitly:
```python
time_limit=min(budget_b, remaining() - 2 * safety_margin - budget_for("D"))
```

This limits Phase B to `min(108, 176.3 - 60 - 54)` = 62.3s, guaranteeing Phase D receives
at least 54s.

### Expected v0.13 timeline (verified against actual run)

```

Phase B time_limit:             62.3s (D reserved)

→ Phase D gets real budget: 4 probes, 3 new sigs
```
